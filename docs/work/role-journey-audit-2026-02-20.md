# Аудит ролей, CJM, воронка и roadmap

Дата аудита: 2026-02-20  
Стенд: `http://127.0.0.1:8080`

## 1. Что проверено
- Матрица доступа по ролям: `admin`, `methodist`, `teacher`, `student`, `anon`.
- Сквозной E2E-сценарий: `student -> methodist -> teacher -> methodist`.
- Ключевые формы и POST-операции с CSRF.
- Проверка ошибок/ограничений (валидация и access control).
- Проверка поведения на неизвестных маршрутах и редакторах сущностей.

## 2. Карта ролей и маршрутов (фактическая)

### Public / anon
- Доступно: `/`, `/programs`, `/about`, `/login`, `/register`, `/consultation`.
- Закрыто редиректом на логин: `/admin/*`, `/methodist/*`, `/teacher/*`, `/student/*`, `/applications`, `/payments`, `/assessments`, `/certificates`.

### Student
- Основной маршрут: `/dashboard -> /student/cabinet`.
- Рабочие зоны: `/student/cabinet`, `/student/profile`, `/applications`, `/applications/new`, `/student/payments/{id}/checkout`.
- Запрещено: `/admin/*`, `/methodist/*`, `/teacher/*`, backoffice-разделы.

### Methodist
- Основной маршрут: `/dashboard -> /methodist/queue`.
- Рабочие зоны: `/methodist/queue`, `/programs`, `/applications`, `/payments`, `/certificates`.
- Запрещено: teacher/student кабинеты и `/assessments`.

### Teacher
- Основной маршрут: `/dashboard -> /teacher/groups`.
- Рабочие зоны: `/teacher/groups`, `/teacher/groups/{programId}`, `/assessments`, `/programs`.
- Запрещено: `/methodist/*`, `/admin/*`, `/payments`, `/certificates`, `/applications`.

### Admin
- Основной маршрут: `/dashboard -> /admin/dashboard`.
- Рабочие зоны: практически все backoffice-разделы.
- Примечание: admin может заходить в `/methodist/queue` и `/teacher/groups` (по текущей политике доступа это разрешено).

## 3. Сквозной сценарий (E2E) и результат
Проверенный поток:
1. Student создал новую заявку (`/applications/new` -> `/applications/save`).
2. Methodist одобрил документы по заявке (`/methodist/queue/{id}/docs`, `APPROVED`).
3. Methodist назначил преподавателя (`/methodist/queue/{id}/assign-teacher`).
4. Methodist провел оплату (`/payments/save`, `PAID`).
5. Methodist открыл полный доступ (`/methodist/queue/{id}/open-full-access`).
6. Teacher выставил аттестацию (`/assessments/save`, score >= 75).
7. Teacher завершил обучение (`/teacher/groups/{programId}/applications/{id}/complete`).
8. Methodist выдал удостоверение (`/certificates/save`).
9. Student видит удостоверение в `/student/cabinet?view=completed`.

Итог: основной бизнес-поток рабочий, включая переходы статусов и выдачу удостоверения.

## 4. Воронка (как в системе сейчас)
По данным `admin/dashboard` на момент аудита:
- Документы: `0`
- Передача и доступ: `1`
- Trial: `1`
- Обучение: `1`
- Удостоверения (готово): `2`

Текущая продуктовая воронка читается как:
`Регистрация/вход -> Подача заявки -> Одобрение документов -> Оплата/доступ -> Обучение -> Аттестация -> Завершение преподавателем -> Выдача удостоверения`.

## 5. CJM (сжатая версия)

### Student CJM
- Цель: попасть на программу и получить удостоверение.
- Шаги: регистрация -> заявка -> ожидание проверки -> оплата -> обучение -> аттестация -> получение удостоверения.
- Болевые точки:
  - по умолчанию в кабинете открывается `active`, завершенные кейсы и сертификаты видны в `view=completed` (неочевидно для нового пользователя).

### Methodist CJM
- Цель: провести слушателя через операционный контур.
- Шаги: проверка документов -> назначение преподавателя -> контроль оплаты/доступа -> контроль очереди на сертификаты -> выдача.
- Болевые точки:
  - ручное “раскидывание” между очередями, нет явного wizard-процесса.

### Teacher CJM
- Цель: вести группу, фиксировать прогресс, завершить обучение.
- Шаги: группы -> карточка группы -> прогресс -> аттестация -> завершение.
- Болевые точки:
  - при попытке завершить без PASS приходит ошибка (это корректно), но пользовательский путь можно подсветить явнее в UI.

### Admin CJM
- Цель: видеть контрольные KPI и узкие места.
- Шаги: dashboard -> drilldown -> переход в операционные разделы.
- Болевые точки:
  - нет единого action-center для инцидентов/просрочек, drilldown в основном обзорный.

## 6. Найденные дыры/дефекты

###[P1] Редактирование заявок недоступно (500)
- Симптом: `GET /applications/{id}/edit` возвращает `500` для методиста и админа.
- Воспроизведение:
  1. Войти как `methodist` или `admin`.
  2. Открыть `/applications/1/edit` (или любой ID).
  3. Получить страницу `Ошибка` (`500`).
- Риск: backoffice-редактирование заявок фактически сломано.

###[P1] Неизвестный путь для авторизованного пользователя уходит в 500 вместо 404
- Симптом: `GET /no-such-page` (в авторизованной сессии) возвращает `500` и `error/general`, а не `404`.
- Риск: не выполняется ожидаемый UX по `404`, сложнее диагностика и хуже контроль нештатных сценариев.

###[P2] Поля `input type="date"` получают значение в формате `dd.MM.yyyy`
- Факт из форм:
  - `paymentDate=20.02.2026`
  - `issueDate=20.02.2026`
  - `resultDate=20.02.2026`
- Риск: для `type="date"` браузерно ожидается `yyyy-MM-dd`; возможны пустые значения/UX-сбои в части браузеров.

## 7. Security/изоляция данных
- Проверено: новый студент не может оплатить чужую заявку (`/student/payments/1/pay` -> `403`), checkout чужой заявки -> `400`.
- Проверено: преподаватель не может редактировать чужие аттестации (`400` с сообщением про свои результаты).
- Вывод: базовая ролевка и изоляция пользовательских данных работают корректно.

## 8. Roadmap (P0/P1/P2)

### P0 (до защиты)
1. Починить `GET /applications/{id}/edit` (блокер для backoffice-сценария).
2. Вернуть корректный `404` для неизвестных URL в авторизованных сессиях.
3. Привести значения date-полей к формату `yyyy-MM-dd` в HTML формах.

### P1 (после стабилизации)
1. Добавить явные системные подсказки по next-step в teacher/methodist потоках (не только ошибка “нельзя”).
2. Улучшить видимость завершенных кейсов для student (переключатель `active/completed` сделать заметнее).
3. Добавить smoke-интеграционные тесты на критический E2E-флоу ролей.

### P2 (продуктовое развитие)
1. Единый action-center для операционных “узких мест” и просрочек.
2. SLA/таймлайны этапов (сколько заявка висит в каждом статусе).
3. Ролевые notification-фиды (что требует действия прямо сейчас).

## 9. Что изменилось на стенде в ходе аудита
В процессе тестов созданы тестовые записи:
- новая заявка (student),
- тестовая оплата,
- тестовая аттестация,
- удостоверение с номером вида `AUTO-CERT-*`,
- несколько тестовых пользователей `aud*`/`user*`.

Если нужен “чистый” стенд для демо, перед защитой стоит перезапустить с чистой in-memory БД.

## 10. Fixed (2026-02-20)
- `[P1]` Исправлен `500` на `GET /applications/{id}/edit` для `admin/methodist`.
  Причина: в `applications/form.html` было небезопасное выражение `application.status.label` без проверки `null`.
  Решение: добавлен null-safe вывод статуса в шаблоне.
- `[P1]` Исправлена обработка несуществующих URL в авторизованной сессии:
  `GET /no-such-page` теперь отдает `HTTP 404` и рендерит `error/404`.
  Решение: добавлен отдельный хендлер для `NoHandlerFoundException`/`NoResourceFoundException` в `GlobalExceptionHandler`.
- `[P2]` Приведены date-поля к ISO-формату для `input type="date"` (`yyyy-MM-dd`) через модель:
  - `Application.applicationDate`
  - `Payment.paymentDate`
  - `AssessmentResult.resultDate`
  - `Certificate.issueDate`
  Решение: добавлены `@DateTimeFormat(iso = DateTimeFormat.ISO.DATE)`.

Проверка после фиксов:
- `GET /applications/1/edit`:
  - `admin` -> `200`
  - `methodist` -> `200`
- `GET /no-such-page` (авторизованный `admin`) -> `404`, шаблон `error/404`.
- Date value в формах:
  - `/payments/new`, `/payments/1/edit`
  - `/assessments/new`, `/assessments/1/edit`
  - `/certificates/new`, `/certificates/1/edit`
  - `/applications/1/edit`
  Во всех случаях `value` в формате `yyyy-MM-dd`.
- Smoke по сохранению:
  - заявки (`student` create + `admin` edit) -> `302`
  - оплаты (new/edit) -> `302`
  - аттестации (new/edit) -> `302`
  - удостоверения (new/edit) -> `302`
