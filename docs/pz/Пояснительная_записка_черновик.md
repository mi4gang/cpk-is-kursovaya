# Пояснительная записка к курсовой работе

## Тема
Проектирование и реализация информационной системы центра повышения квалификации.

## Введение
Центры повышения квалификации работают с большим количеством однотипных операций: формирование программ обучения, прием и обработка заявок слушателей, учет оплат, фиксация результатов аттестации и выдача удостоверений. При отсутствии единой информационной системы данные часто ведутся в разрозненных таблицах, что приводит к задержкам в обработке заявок, ошибкам при учете платежей и сложностям в подготовке отчетности.

Актуальность темы обусловлена необходимостью повысить прозрачность и управляемость бизнес-процессов центра повышения квалификации. Для этого в рамках курсовой работы спроектирована и реализована учебная информационная система, которая автоматизирует ключевой линейный процесс: "Программа -> Заявка -> Оплата -> Аттестация -> Удостоверение".

Цель работы — разработка информационной системы, обеспечивающей учет основных сущностей центра повышения квалификации, разграничение доступа по ролям и контроль прохождения слушателя по учебному циклу.

Задачи работы:
1. Выполнить моделирование предметной области в нотациях IDEF0, IDEF3, DFD, UML и IDEF1X.
2. Сформировать функциональные и нефункциональные требования к системе.
3. Спроектировать реляционную структуру данных и обосновать приведение к 3НФ.
4. Реализовать MVP-прототип на Java + Spring Boot + PostgreSQL + Thymeleaf.
5. Обеспечить устойчивость к ошибкам и базовые механизмы безопасности.

Объект исследования: процессы управления обучением в центре повышения квалификации.

Предмет исследования: методы и средства проектирования информационной системы для поддержки процессов зачисления и обучения слушателей.

## 1. Теоретическая часть
### 1.1 Подход к проектированию
Для формализации предметной области использован набор нотаций, рекомендованный методическими указаниями:
- IDEF0 для функциональной декомпозиции процессов и выделения входов, выходов, управляющих воздействий и механизмов.
- IDEF3 для описания последовательности выполнения операций.
- DFD для отображения потоков данных между процессами, внешними сущностями и хранилищами.
- UML (Use Case, Activity, Sequence) для представления пользовательских сценариев и взаимодействия объектов.
- IDEF1X для моделирования структуры данных реляционной БД.

Комплексное применение этих нотаций позволяет избежать логических разрывов между аналитикой, проектированием и реализацией.

### 1.2 Выбор технологического стека
Выбран стек Java + Spring Boot + PostgreSQL + Thymeleaf. Данный набор технологий принят как стабильный и соответствующий промышленным стандартам для обеспечения безопасности бизнес-процессов и управляемости решения в учебном проекте.

Обоснование:
- Spring Boot ускоряет создание серверной части и обеспечивает предсказуемую архитектуру.
- Spring Data JPA (Hibernate) упрощает доступ к данным и снижает риск ошибок при работе с SQL.
- Spring Security обеспечивает авторизацию и ролевой доступ.
- Thymeleaf дает возможность быстро реализовать веб-интерфейс без усложнения фронтенд-части.
- PostgreSQL подходит для хранения транзакционных данных и поддерживает надежную реляционную модель.

## 2. Аналитическая часть
### 2.1 Описание предметной области
Центр повышения квалификации предоставляет образовательные программы для специалистов. Основные участники процесса:
- Администратор.
- Методист.
- Преподаватель.
- Слушатель.

Ключевые бизнес-процессы:
1. Формирование и публикация программ обучения.
2. Подача заявок слушателями.
3. Проверка и учет оплаты.
4. Проведение аттестации и внесение результата.
5. Выдача удостоверения об успешном завершении.

### 2.2 Проблемы AS-IS
В ручном режиме обнаруживаются следующие проблемы:
- Дублирование данных в разных файлах.
- Потеря актуальности статусов заявки.
- Ошибки при сверке оплат.
- Сложность подготовки итоговой отчетности.
- Отсутствие централизованного разграничения доступа.

### 2.3 Целевая модель TO-BE
Целевая модель предусматривает единый источник данных, веб-доступ по ролям и формализованный маршрут прохождения слушателя. Все действия фиксируются в БД и доступны для анализа на главной панели.

### 2.4 Функциональные требования
1. Регистрация и авторизация пользователей.
2. Разделение ролей и ограничение доступа к функциям.
3. CRUD для программ, заявок, оплат, результатов аттестации, удостоверений.
4. Поиск по названию программы и ФИО слушателя.
5. Сортировка записей по ключевым полям.
6. Отображение статистики на главной странице.
7. Отдельная страница "Об авторе".
8. Корректная обработка ошибок без аварийного завершения.

### 2.5 Нефункциональные требования
1. Трехзвенная архитектура: веб-интерфейс, сервер приложения, БД.
2. Поддержка безопасной работы с пользовательским вводом.
3. Устойчивость при некорректных данных и ошибках маршрутизации.
4. Простота сопровождения и расширения системы.
5. Достаточная скорость обработки типовых операций CRUD.

### 2.6 Ограничения
1. Проект реализуется в учебном формате (MVP).
2. Внешние интеграции (платежные шлюзы, госреестры) не выполняются.
3. Основной упор делается на прозрачность модели и соответствие критериям курсовой работы.

## 3. Проектная часть
### 3.1 IDEF0
В рамках работы разработаны:
- контекстная диаграмма A-0,
- диаграмма уровня A0,
- декомпозиция ключевого процесса (обработка заявки) до подуровня.

На диаграммах определены входы (данные программ и слушателей), выходы (статусы и удостоверения), управляющие воздействия (регламенты, учебный план), механизмы (роли сотрудников и ИС).

### 3.2 IDEF3
IDEF3 отражает линейную последовательность процесса:
1. Регистрация программы.
2. Подача заявки.
3. Оплата.
4. Аттестация.
5. Выдача удостоверения.

### 3.3 DFD
На DFD показаны:
- внешние сущности: слушатель, методист,
- процессы: обработка заявки и учет оплат,
- хранилища: "Заявки", "Платежи",
- потоки данных между участниками и хранилищами.

### 3.4 UML Use Case
Для каждой роли определены варианты использования:
- Администратор: управление ключевыми объектами системы.
- Методист: программы, заявки, оплаты, удостоверения.
- Преподаватель: внесение результатов аттестации.
- Слушатель: подача заявки.

### 3.5 UML Activity
Диаграмма активности моделирует полный цикл прохождения слушателя от публикации программы до выдачи удостоверения с контрольной точкой оплаты.

### 3.6 UML Sequence
Диаграмма последовательности описывает сценарий создания заявки: взаимодействие слушателя, веб-интерфейса, контроллера и БД.

### 3.7 IDEF1X и структура БД
Схема данных включает сущности:
- User,
- Program,
- Application,
- Payment,
- AssessmentResult,
- Certificate.

Связи:
- Program 1:N Application.
- Application 1:1 Payment.
- Application 1:1 AssessmentResult.
- Application 1:1 Certificate.

## 4. Нормализация до 3НФ
Для каждой таблицы выполнено обоснование:

1. Таблица `programs` приведена к 3-й нормальной форме, так как все неключевые атрибуты (title, description, duration_hours, start_date, end_date, active) функционально зависят только от первичного ключа `id`, и транзитивные зависимости отсутствуют.

2. Таблица `applications` приведена к 3-й нормальной форме, так как неключевые атрибуты (listener_full_name, listener_email, listener_phone, application_date, status, program_id) зависят только от `id`, а транзитивные зависимости отсутствуют.

3. Таблица `payments` приведена к 3-й нормальной форме, так как атрибуты (amount, payment_date, status, application_id) зависят от `id` и не зависят транзитивно от других неключевых полей.

4. Таблица `assessment_results` приведена к 3-й нормальной форме, так как атрибуты (score, status, result_date, comment, application_id) зависят только от `id`, транзитивные зависимости отсутствуют.

5. Таблица `certificates` приведена к 3-й нормальной форме, так как атрибуты (number, issue_date, status, application_id) зависят от первичного ключа `id`, а транзитивные зависимости отсутствуют.

6. Таблица `users` приведена к 3-й нормальной форме, так как атрибуты (username, password, full_name, email, phone, role, enabled) зависят только от `id`, и транзитивные зависимости отсутствуют.

## 5. Описание реализации программы
### 5.1 Архитектура
Реализована трехзвенная архитектура:
1. Уровень представления: Thymeleaf-шаблоны.
2. Серверный уровень: Spring Boot контроллеры и бизнес-логика.
3. Уровень данных: PostgreSQL + JPA-репозитории.

### 5.2 Модуль безопасности
Используется Spring Security с формой входа и ролевой моделью доступа.

Роли:
- ROLE_ADMIN
- ROLE_METHODIST
- ROLE_TEACHER
- ROLE_STUDENT

Доступ к критичным операциям (создание/редактирование/удаление) ограничен аннотациями `@PreAuthorize`.

### 5.3 Устойчивость к ошибкам
Реализована глобальная обработка исключений через `@ControllerAdvice`.

При ошибках валидации, некорректных параметрах, 404 и 500 пользователь получает понятную страницу с сообщением:
"Произошла ошибка. Проверьте данные и вернитесь назад".

Это обеспечивает выполнение требования по отсутствию аварийного завершения и повышает предсказуемость интерфейса.

### 5.4 Обоснование безопасности
В проекте применяются средства безопасности по умолчанию:
- ORM (Hibernate) снижает риск SQL-инъекций за счет параметризованных запросов.
- Шаблонизатор Thymeleaf по умолчанию экранирует HTML-вывод и снижает риск XSS.
- Spring Security обеспечивает контроль доступа к функционалу по ролям.

### 5.5 Реализованный функционал MVP
1. Авторизация и доступ по ролям.
2. CRUD по ключевым сущностям.
3. Поиск и сортировка в списках.
4. Статистика на главной странице:
- общее количество слушателей,
- количество активных программ,
- сумма оплаченных платежей.
5. Страница "Об авторе" согласно требованиям.

## 6. Тестирование и проверка сценариев
Проверены основные сценарии приемки:
1. Неавторизованный пользователь не имеет доступа к функционалу.
2. Пользователь каждой роли видит только доступные ему действия.
3. Все CRUD-операции работают корректно в пределах MVP.
4. Поиск и сортировка возвращают ожидаемые наборы данных.
5. Ошибки ввода приводят к пользовательским сообщениям, а не к падению приложения.
6. Статистика на главной странице соответствует данным БД.
7. Сквозной сценарий "программа -> заявка -> оплата -> аттестация -> удостоверение" проходит полностью.

## 7. Заключение
В рамках курсовой работы выполнено проектирование и реализация MVP информационной системы центра повышения квалификации. Проект покрывает обязательные функциональные требования, содержит комплект диаграмм по методическим указаниям и демонстрирует практическое применение трехзвенной архитектуры.

Выбранный стек технологий обеспечивает надежную основу для учебного и практического применения. Система решает базовые бизнес-задачи учета, снижает количество ручных операций и формирует прозрачный маршрут прохождения слушателя от заявки до выдачи удостоверения.

Полученный результат соответствует заявленным целям, может быть использован как основа для дальнейшего расширения функциональности и подготовлен к демонстрации на защите.
